## Problem Statement

When a reviewer receives a pile of changes from a model, they have to do unguided work before
any real reviewing happens: figure out where to start, decide what order makes sense (usually
general → specific), and reconstruct what each file does before reading its diff. This is slow,
easy to get wrong, and the reviewer often reads files in a confusing order and loses the thread.
Meanwhile Galley's right sidebar pane sits empty.

## Solution

An AI-guided review that turns a pile of changes into a flow. Galley already reviews **one
file at a time** (the diff swaps as you select a file) and keeps the **full file tree** on the
left; guided review layers on top of that without changing the diff (per-block accept/reject,
line comments, the Viewed button all stay). The guidance lives in two new surfaces — an
**overview page** and a **top guide bar** above the diff — decided via the prototypes in
`prototypes/guided-review-v4.html` / `v5.html` ("count + fill" variant):

- An **Overview page** (its own screen, reachable anytime via a ⌂ control): a **concise
  overall overview** of what the changeset does, the **PR description** optionally (PR mode),
  the **category plan**, and a **Start guided review →** action that enters the per-file flow.
- A **top guide bar** above the diff for the current file: its **change category** (a semantic
  label like Config/Core/Wiring/Routes/Tests/Docs, distinct from the folder), the file name,
  a **concise per-file summary** shown *before* the diff, a **"why flagged"** note for critical
  files, the navigation actions `[⌂ · ← Prev · Next →]` (far-left), and minimal progress.
- Files are **ordered general → specific**; **Next/Prev walk that order** (the tree still lets
  you jump anywhere), so the reviewer forms a mental model, reads the diff, then advances.
- A **category stepper** (count + fill): each category segment shows `n/m` and a fill bar that
  grows as its files are viewed and turns green at `m/m` — the macro-progress of the review. A
  multi-file category (e.g. 5 Wiring files) is `n/5` and walked file-by-file; the tree shows
  the exact file.

Because the desk runs no model, the guided review is **generated by the coding agent** (which
already has model access and the diff) and **attached** to the review, the same way agent
comments are. The desk renders it. Applies to repo (local git) and PR review modes; file mode
is single-file so the guide is not applicable.

## User Stories

1. As a reviewer, I want a one-paragraph overview of the whole changeset, so that I know what
   I'm about to review before diving in.
2. As a reviewer, I want the files ordered general → specific, so that I build context before
   hitting the details.
3. As a reviewer, I want a concise summary of each file's changes shown before its diff, so
   that I have a mental model of what to expect.
4. As a reviewer, I want to read the summary, then the diff, then move to the next file in
   order, so that I have a guided flow instead of wandering.
5. As a reviewer, I want the most critical changes highlighted, so that I spend my attention
   where it matters.
6. As a reviewer reviewing a PR, I want to optionally see the PR description in the pane, so
   that I have the author's intent alongside the changes.
7. As a reviewer, I want to click a file in the guided list and jump to its diff, so that the
   guide drives navigation.
8. As a reviewer, I want my place in the guided flow tracked (what I've reviewed), so that I
   can resume where I left off.
9. As a coding agent, I want to generate the guided review from the diff and attach it, so that
   the desk stays model-free and the guidance reflects my understanding of the change.
10. As a coding agent, I want to regenerate the guide after a reload, so that the guidance
    matches the current diff.
11. As a reviewer, I want the guide to degrade gracefully when absent, so that the desk is
    fully usable before any guide is attached.
12. As a reviewer, I want the per-file summary to be genuinely concise, so that reading it is
    faster than reading the diff cold.
13. As a reviewer, I want critical-change highlights to map to specific files/hunks, so that I
    can act on them, not just read a warning.

## Implementation Decisions

- **Guided review is agent-generated and attached.** Consistent with the "no model in the
  desk" decision (see the project README and the comment mechanism): the agent computes the
  guide and attaches it; the desk only renders. This keeps the desk model-free and the
  guidance honest to the agent's understanding.
- **Data shape.** `ReviewState` gains an optional `guide`: an overview string, an optional PR
  description, and an ordered list of per-file entries `{ path, order, category, summary,
  critical?, why? }`. `order` drives Next/Prev; `category` is the stepper grouping; `critical`
  + `why` drive the flag and "why flagged" note. Absent `guide` → no guide surfaces render and
  the desk works exactly as today.
- **Attach path.** A `POST /api/guide` endpoint and a `galley guide` subcommand mirror the
  comment flow: read the desk lock, POST the guide JSON to the live desk (falls back to
  appending to the saved state if no desk is live). The guide survives reload via the merge.
- **Rendering.** No right pane. The **Overview page** renders overview → optional PR
  description → category plan → Start. The **top guide bar** renders, for the current file, the
  category chip + filename + summary (+ "why flagged" for critical) + nav actions + the
  category stepper (count + fill) + minimal progress. The center keeps Galley's existing
  single-file diff (selecting a file or pressing Next re-renders that one file). Reviewed-state
  ties into the existing `reviewedFiles`, which also feeds the per-category fill.
- **Ordering & summaries are the agent's job.** The general→specific ordering, the per-file
  summaries, and the criticality scoring are produced by the agent's model call; Galley does
  not implement heuristics for them. Galley defines the schema and the rendering.
- **Deep module.** A small "guide" model + validator (schema in/out) and the merge/attach
  logic; rendering is UI.

## Testing Decisions

- A good test asserts the **attach/merge and validation behavior**: a valid guide attaches and
  survives a reload/merge; an invalid guide is rejected; absent guide leaves the state usable.
  It does not test the AI-produced content quality (that's not deterministic).
- **Modules to test:** the guide schema validation and the guide-attach/merge into review
  state (including surviving the reload merge from the live-desk-robustness PRD).
- **Prior art:** the comment-append logic and the state-merge tests; reuse the `node:test`
  setup.

## Out of Scope

- The desk generating the guide itself (no in-desk model).
- Deterministic, Galley-implemented ordering heuristics (the agent decides order/criticality).
- Per-line AI critique inline in the diff (guidance is per-file summaries + critical-hunk
  highlights, not line-by-line annotations).
- Auto-refreshing the guide on a timer.

## Further Notes

This is the feature that most changes how review *feels* — from "here's a wall of diffs" to "a
guided tour." It depends on review modes existing (so the PR description and PR-vs-repo framing
make sense) and on the UI architecture decision, since it adds the overview page + top guide
bar surfaces. The final layout/mechanics were chosen via the prototypes in
`prototypes/guided-review-v4.html` and `v5.html` (the "count + fill" category stepper).
